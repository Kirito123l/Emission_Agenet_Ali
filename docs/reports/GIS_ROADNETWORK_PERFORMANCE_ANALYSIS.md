# GIS 路网计算与可视化性能分析报告

## 原始问题

当前仓库的 agent 已经具备 GIS 可视化能力，可以对宏观路网排放结果进行地图展示。但在真实大规模路网场景下，尤其是路段数量达到上万条时，系统的计算和可视化速度明显变慢。在部署到仅有 2GB 内存的云服务器后，这个问题更加突出。

原始诉求可以概括为：

- 为什么在路网很多时，系统会明显变慢？
- 慢主要发生在哪些阶段，是后端计算、数据传输还是前端渲染？
- 当前实现中，真正限制速度的核心评价指标是什么？

本报告基于代码路径检查、数据规模检查和本地基准测试，对该问题进行完整分析。

## 结论摘要

这不是单点性能问题，而是多段瓶颈叠加：

1. 后端宏观排放计算存在高频全表查找，属于当前最重的 CPU 瓶颈。
2. 前端 Leaflet 对大规模路段采用逐条 polyline 渲染，同时还默认加载整份路网底图，造成浏览器主线程负载过高。
3. 大尺寸 `map_data` 会被完整传输、完整持久化、完整回放，导致网络、磁盘 I/O 和内存占用同步放大。
4. 2GB 云服务器主要放大的是后端计算、JSON 序列化、历史加载和会话持久化成本；浏览器端地图交互卡顿则主要取决于访问端设备本身。

从当前实现看，真正决定速度上限的不是单一“服务器配置”，而是以下四个量：

- 后端 `ms / 1000 links`
- 排放矩阵查找次数
- 地图响应体积（MB）
- 地图总 feature 数 / 总顶点数

## 分析范围

本次分析覆盖以下路径：

- 宏观排放计算：`calculators/macro_emission.py`
- 宏观工具封装与地图数据生成：`tools/macro_emission.py`
- 前端地图渲染：`web/app.js`
- GIS 底图接口：`api/routes.py`
- 会话历史持久化：`api/session.py`
- 结果文件导出：`skills/macro_emission/excel_handler.py`

## 代码级瓶颈分析

### 1. 后端计算瓶颈：宏观排放计算采用逐路段、逐车型、逐污染物循环

`calculators/macro_emission.py` 中的 `calculate()` 会对每个路段逐一计算，随后进入 `_calculate_link()`，再进一步对每个车型和每个污染物执行排放率查询。

关键路径：

- `calculate()`：第 75-98 行
- `_calculate_link()`：第 136-246 行
- `_query_emission_rate()`：第 248-262 行

当前结构是：

```text
for link in links_data:
    for vehicle in fleet_mix:
        for pollutant in pollutants:
            _query_emission_rate(matrix, ...)
```

而 `_query_emission_rate()` 不是 O(1) 查表，而是对整张 pandas DataFrame 做布尔过滤：

```python
result = matrix[
    (matrix[self.COL_OPMODE] == 300) &
    (matrix[self.COL_POLLUTANT] == pollutant_id) &
    (matrix[self.COL_SOURCE_TYPE] == source_type) &
    (matrix[self.COL_MODEL_YEAR] == model_year)
]
```

这意味着：

- 每一次排放率查询，都会扫描整张排放矩阵。
- 路网规模越大，查询次数线性上升。
- 默认车队组成包含 5 个车型。
- 常见输出污染物是 2 个，例如 `CO2` 和 `NOx`。

因此，单条路段默认大约触发：

- `5 车型 × 2 污染物 = 10 次矩阵查询`

对于 `25,370` 条路段，总查询次数约为：

- `25,370 × 10 = 253,700 次`

而夏季排放矩阵实测有：

- `184,574` 行

也就是说，当前实现会触发大规模重复 DataFrame 过滤，这是当前后端最主要的性能瓶颈。

### 2. 后端还有额外 I/O：每次计算都重新读取排放矩阵文件

`_load_emission_matrix()` 每次调用都会重新从 CSV 读取整张矩阵，而不是进程内缓存。

这部分不是最重瓶颈，但会带来额外的磁盘读取和 DataFrame 构建成本，尤其在多次重复计算时会放大。

### 3. 前端渲染瓶颈：每条路段都创建一个 Leaflet polyline

`web/app.js` 的 `initLeafletMap()` 中，对 `mapData.links` 逐条执行：

- 坐标转换
- `L.polyline(...)`
- `bindPopup(...)`
- `addLayer(...)`
- 加入 bounds 计算

这意味着：

- 每条路段都会创建一个独立的 Leaflet 对象。
- 每条路段都会生成 popup HTML。
- 每条路段的数据还会挂载到 `polyline._linkData`。

即使开启了 `preferCanvas: true`，当路段数达到上万条时，浏览器主线程仍然需要承担大量对象创建、样式计算、事件绑定和重绘成本。

### 4. 前端默认额外加载整份路网底图，导致图层数量翻倍

`web/app.js` 中不仅绘制排放图层，还默认加载了 `roadnetwork.geojson` 作为背景路网图层：

- `roadLayerGroup` 默认 `addTo(map)`
- `loadGISRoadNetwork()` 完成后会把整份 GeoJSON 加入该图层

实测数据：

- `static_gis/roadnetwork.geojson` 文件大小约 `6.1 MB`
- `roadnetwork.geojson` 中 feature 数量为 `25,370`
- `basemap.geojson` feature 数量为 `16`

这意味着在完整上海路网测试场景里，一张图的首屏渲染大致同时包含：

- `25,370` 条排放路段
- `25,370` 条背景路网
- `16` 个行政区面

总量接近：

- `50,756` 个矢量要素

这已经超出了普通 Leaflet 页面在低配置环境中的舒适渲染范围。

### 5. 地图数据传输和历史持久化会放大问题

每次有地图结果时，`map_data` 会被：

1. 作为接口响应发回前端
2. 写入 session history
3. 在历史页面重新读回
4. 在历史渲染时重新实例化地图

相关实现：

- `api/routes.py` 会发送 `type: "map"` 的流式消息
- 同一个 `done` 消息里又附带一次 `map_data`
- `api/session.py` 会把完整 `map_data` 直接写入 `_history`
- `SessionManager._save_to_disk()` 会把所有会话历史重新 dump 到磁盘
- `SessionManager._load_from_disk()` 会把历史全部读回内存

这导致三个明显后果：

- 网络 payload 变大
- 磁盘写入变大
- 内存常驻数据变大

### 6. 打开历史会话时，旧地图会被完整回放

`/sessions/{session_id}/history` 会直接返回该会话完整 `_history`。
前端 `renderHistory()` 随后对历史中的每条 assistant 消息执行 `addAssistantMessage()`，如果存在 `map_data`，则再次调用 `renderEmissionMap()`。

因此：

- 单个超大地图消息不只是“生成时慢”
- 重新打开历史会话时，也会再次慢一次

如果一个会话里包含多次大规模地图结果，历史页的加载会进一步恶化。

### 7. 结果导出属于附加成本，不是主瓶颈但会补时间

宏观工具在有输入文件时会自动生成下载结果文件。

这一过程会：

- 再次读取原始 Excel / CSV / ZIP
- 把排放列拼接回去
- 再写出一份新的 Excel

这部分属于额外 I/O，通常不是第一瓶颈，但在 25k 路段场景下会带来额外的秒级开销。

## 数据规模与现场证据

本次检查得到的关键数据如下：

- `static_gis/roadnetwork.geojson`：`6,148,993 bytes`
- `static_gis/basemap.geojson`：`69,199 bytes`
- `roadnetwork.geojson` feature 数量：`25,370`
- `basemap.geojson` feature 数量：`16`
- `data/sessions` 当前总大小：约 `266 MB`
- 单个历史 JSON 最大值：约 `30 MB`
- `test_data/test_shanghai_allroads.xlsx` 文件大小：约 `3.5 MB`
- 夏季宏观排放矩阵行数：`184,574`

这些数字说明：

- 真正庞大的不是输入文件本身，而是计算结果和地图历史数据。
- 路网底图本身就已经是大对象，再叠加排放结果图层后，前端和后端都会有明显压力。

## 基准测试结果

### 1. 核心计算器基准

直接对 `MacroEmissionCalculator` 进行基准测试，得到：

- `100` 条路段：约 `0.875s`
- `1000` 条路段：约 `9.115s`

这说明：

- 计算成本基本随路段数量线性增长
- 单条路段的计算成本本身并不低
- 当规模扩大到 `25,370` 条时，单纯后端计算就会变得很重

### 2. 地图数据构建基准

对 `_build_map_data()` 的测试结果：

- `100` 条路段：约 `0.002s`
- `1000` 条路段：约 `0.016s`

这说明：

- 后端“把结果组织成 map_data”本身并不是最主要瓶颈
- 真正慢的是：
  - 前面的排放率计算
  - 后面的前端渲染
  - 中间的大 JSON 传输与持久化

### 3. 完整工具链测试现象

对 `test_shanghai_allroads.xlsx` 直接执行完整宏观工具链时，当前环境下在 `100s+` 内仍未完成。

这说明：

- 单看小样本还不够
- 全量路网时，后端实际体感已经进入“明显不可接受”的级别

## 为什么 2GB 云服务器会更明显

需要区分两个维度：

### 1. 服务器配置影响的部分

2GB 服务器会直接影响：

- pandas / openpyxl / JSON 序列化时的峰值内存
- 会话历史加载后的常驻内存
- 大对象写盘和读盘时的缓存能力
- 单线程 Python 计算的整体吞吐

因此，以下现象会更严重：

- 首次请求慢
- 大文件计算慢
- 历史会话切换慢
- 多用户情况下容易抖动

### 2. 服务器配置不直接决定的部分

地图平移、缩放、图层切换是否顺滑，主要还是浏览器端渲染问题。

也就是说：

- 如果用户是在本地电脑浏览网页，那么服务器内存并不直接决定地图拖动是否卡顿。
- 服务器主要决定“结果多久算出来、多久发过来、历史多久加载出来”。
- 浏览器设备决定“地图拿到以后能不能顺滑画出来”。

## 当前实现中真正限制速度的评价指标

如果要给这个问题建立一套明确的性能评价指标，建议优先看以下几项。

### A. 后端计算指标

1. `calc_ms_per_1000_links`

- 定义：每 1000 条路段所需的纯计算时间
- 用途：衡量宏观排放计算核心算法是否可扩展

2. `matrix_lookup_count`

- 定义：`路段数 × 车型数 × 污染物数`
- 用途：衡量当前排放矩阵查找压力

3. `peak_rss_mb`

- 定义：计算过程峰值常驻内存
- 用途：衡量 2GB 机器是否有 OOM 或频繁 GC 风险

### B. 网络与序列化指标

4. `map_payload_mb`

- 定义：单次地图响应 JSON 大小
- 用途：衡量网络传输和浏览器 JSON 解析成本

5. `history_file_mb`

- 定义：单会话历史文件大小
- 用途：衡量历史读写对磁盘和内存的压力

6. `history_save_ms`

- 定义：一次 `save_turn + dump` 的时间
- 用途：衡量会话持久化是否成为明显拖点

### C. 前端渲染指标

7. `vector_feature_count`

- 定义：地图总 feature 数
- 用途：衡量 Leaflet 首屏和交互负载

8. `vertex_count`

- 定义：所有 geometry 坐标点总数
- 用途：衡量实际绘制复杂度

9. `first_map_render_ms`

- 定义：从收到 `map_data` 到地图首屏可见的耗时
- 用途：衡量用户体感

10. `pan_zoom_fps`

- 定义：拖拽、缩放操作的帧率
- 用途：衡量地图是否真正“可用”

## 当前最核心的三个限制因子

如果只保留三个最关键指标，那么当前系统的速度上限主要由以下三个量决定：

1. `calc_ms_per_1000_links`
2. `map_payload_mb`
3. `vector_feature_count`

换句话说：

- 后端算得快不快，决定结果多久能出来
- 返回包大不大，决定结果多久能到浏览器
- 地图对象多不多，决定结果到了以后能不能画得动

## 优先级排序

基于当前实现，建议将瓶颈优先级排序为：

1. 宏观排放矩阵查找方式
2. 前端 5 万级矢量要素渲染
3. 历史持久化的大 JSON 读写
4. 结果导出带来的附加 I/O

## 后续优化方向

本报告只做分析，不直接修改实现。但从工程收益看，后续应优先考虑以下方向：

### 1. 优化排放矩阵查找

目标：

- 将 `_query_emission_rate()` 从 DataFrame 过滤改为预索引字典查表

预期收益：

- 直接降低后端核心 CPU 开销
- 属于收益最高的单点优化

### 2. 降低前端地图要素数量

目标：

- 大规模路网时默认不加载整份背景路网
- 或按缩放级别分层加载
- 或做抽稀 / 聚合 / 瓦片化

预期收益：

- 显著降低首屏渲染时间
- 改善缩放和平移流畅度

### 3. 减少历史消息中的大对象持久化

目标：

- 历史里不保存完整 `map_data`
- 或只保存摘要、采样结果、文件引用

预期收益：

- 降低磁盘 I/O
- 降低内存占用
- 减少历史页面加载成本

### 4. 避免重复发送地图数据

目标：

- 流式接口不要在 `done` 消息里再次附带完整 `map_data`

预期收益：

- 降低网络开销
- 降低前后端序列化成本

## 结语

当前系统在“小规模路网 + 单次结果展示”场景下可以工作，但在“完整城市路网 + 历史保存 + 在线地图回放”这个场景下，已经暴露出明确的扩展性瓶颈。

因此，这个问题的本质不是“Leaflet 有点慢”或者“云服务器只有 2GB”，而是：

- 后端算法复杂度偏高
- 前端默认绘制对象过多
- 数据生命周期过长，导致同一份大地图被反复传输、保存和回放

如果后续进入优化阶段，应该优先处理：

1. 排放矩阵查找方式
2. 大规模地图渲染策略
3. 会话历史中的大对象管理

这三项处理完，整体体感性能会有最明显改善。
